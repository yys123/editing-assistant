# 指南切片确认层设计

## 目标

在内容质量评审和 AI 整合两个优先环节中，引入统一的“指南切片确认层”。系统先根据当前任务从已上传或已接入的指南数据中筛选相关切片，再交给用户确认；最终下游 AI 只使用用户确认过的切片，而不是直接接收整篇指南全文。用户需求分析暂时不强制接入指南切片，只保留后续扩展入口。

该设计的目标是：

- 降低长指南全文进入模型导致的上下文压力和信息噪声。
- 让用户在关键 AI 分析前明确确认证据范围。
- 提高内容质量评审和 AI 整合的证据可追溯性，并为后续用户需求分析接入指南证据保留空间。
- 为后续升级向量库或混合检索保留架构入口。

## 背景

当前系统已经支持上传参考数据源，也支持从指南库检索并添加指南详情。指南进入流程后会被作为 `ReferenceDoc` 或 `ReferenceInput` 使用，后续质量评审和 AI 整合再由后端拼接 prompt。

现有代码中已经有部分相关能力：

- AI 整合中，`backend/services/generator.py` 已有 `_build_reference_chunks()` 和 `_select_reference_chunks()`，可以把参考资料切片并按任务相关性选择片段。
- 内容质量评审中，`backend/services/analyzer.py` 已有 `_extract_relevant_reference_chunks()` 和 `_build_reference_block()`，可以按章节标题和章节正文筛选相关参考片段。
- 重点指南主证据设计已经把“重点指南”从普通提醒升级为主证据来源，但用户仍无法在最终调用前确认具体使用了哪些指南片段。

现有缺口是：切片选择仍是后端内部的隐式行为，用户看不到候选片段，无法确认、排除或补选；不同环节也没有统一的切片模型和确认结果。

## 范围

包含：

- 为上传指南和指南库详情建立统一切片模型。
- 在内容质量评审和 AI 整合中加入“候选切片 -> 用户确认 -> 下游 AI”的流程。
- 为用户需求分析预留切片确认能力，但第一版暂不接入。
- 复用轻量检索作为第一版召回方案。
- 保留向量库升级入口，但第一版不强制引入向量数据库。
- 保存每次任务确认过的切片，用于历史记录和证据追溯。

不包含：

- 自动判断哪篇指南一定应被用户采用。
- 自动从外部网络更新最新版指南。
- 自动替代用户对证据的最终确认。
- 第一版不建设独立大规模医学知识库。
- 第一版不做复杂医学语义冲突判定。

## 推荐方案

采用“统一切片确认层 + 轻量检索优先 + 可升级向量库”的方案。

核心流程：

1. 指南进入参考数据源后，后端将其切成结构化片段。
2. 内容质量评审和 AI 整合任务生成检索 query。
3. 后端根据 query 召回候选切片。
4. 前端展示候选切片，用户确认、排除或补选。
5. 下游 AI 只接收确认后的切片。
6. 历史记录保存本次确认切片和来源信息。

这样可以先获得最大的产品收益：可控、透明、减少上下文浪费；同时避免第一步就引入较重的向量库基础设施。

## 统一数据模型

建议新增后端模型：

```py
class ReferenceChunk(BaseModel):
    chunk_id: str
    source_id: int
    source_filename: str
    title_path: str = ""
    text: str
    context_before: str = ""
    context_after: str = ""
    paragraph_index: int = 0
    source_ref_ids: List[str] = []
    score: float = 0.0
    reason: str = ""
```

说明：

- `chunk_id` 使用稳定编号，例如 `R2-C003`，表示参考数据源 2 的第 3 个切片。
- `source_id` 沿用上传后的全局参考数据源编号，确保引用标记不混乱。
- `title_path` 保存指南章节路径，例如“诊断 / 实验室检查”。
- `context_before` 和 `context_after` 用于用户查看上下文，不一定进入最终 prompt。
- `source_ref_ids` 保存源内参考文献号，继续支持 `[2-15]` 这类引用。
- `score` 和 `reason` 用于前端解释为什么召回该切片。

确认结果可以使用：

```py
class ConfirmedReferenceChunk(BaseModel):
    chunk_id: str
    source_id: int
    source_filename: str
    title_path: str = ""
    text: str
    source_ref_ids: List[str] = []
    selected_by: str = "user"
```

## 环节数据流

### 内容质量评审

内容质量评审按章节进行，因此切片召回应围绕当前章节构造 query。

建议 query 组成：

- 疾病名。
- 当前章节标题。
- 当前章节正文摘要或前若干字符。
- 内容质量标准中的关键维度。
- 用户已选择的重点指南信息。

流程：

1. 用户进入某章节质量评审。
2. 系统根据章节内容召回候选指南切片。
3. 前端展示“本章节相关指南片段”确认区。
4. 用户确认后发起章节评审。
5. 后端评审 prompt 中只注入确认切片。
6. 评审输出中的 `guideline_evidence` 必须来自确认切片。

收益：

- 减少因整篇指南过长导致的模型漏读。
- 避免模型从未经用户确认的指南片段中找依据。
- 用户可以在审评前排除明显无关或不适用的指南片段。

### 用户需求分析（暂缓接入）

用户需求分析通常围绕 Q&A 聚类、用户主题和词条章节覆盖情况展开。这个环节目前主要依赖用户问题、词条原文和章节映射，短期内不一定需要指南切片参与。

第一版建议暂缓在用户需求分析中加入指南切片确认，原因是：

- 用户需求分析的核心目标是识别用户关心什么、词条是否覆盖，而不是直接做循证修订。
- 如果每个需求主题都要求用户确认指南切片，操作成本会明显增加。
- 需求分析已有 Q&A 聚类和章节映射流程，贸然引入指南证据可能让问题边界变重。
- 质量评审和 AI 整合对指南证据的依赖更直接，优先接入收益更高。

后续如果需要接入，可以按需求主题构造 query：

- 疾病名。
- 需求簇主题。
- 代表性用户问题。
- 相关词条章节标题。
- 已识别的覆盖缺口。

预留流程：

1. 系统完成用户问题聚类或需求主题识别。
2. 对需要指南支撑的需求主题召回候选指南切片。
3. 用户确认该主题可参考的指南片段。
4. AI 再判断该需求是否应进入内容修订范围，或作为后续 AI 整合的证据输入。
5. 需求分析结果可选保存关联的确认切片。

暂缓收益：

- 第一版减少用户确认步骤，把精力集中在最需要指南证据的质量评审和 AI 整合。
- 保持用户需求分析的轻量和快速。
- 后续仍可通过相同切片模型平滑接入，不需要推倒重做。

### AI 整合

AI 整合是最终修订输出环节，应严格只使用确认过的证据切片。

建议 query 组成：

- 疾病名。
- 用户输入的整合要求。
- 已确认的质量评审问题。
- 已确认的用户需求缺口（如已有）。
- 用户选择的原词条章节内容。
- 重点指南编号。

流程：

1. 用户选择整合问题、原词条范围和参考资料。
2. 系统召回候选指南切片。
3. 用户确认最终可用于本次整合的切片。
4. 后端构造“重点指南主证据区”和“普通参考资料补充区”。
5. 最终 AI 只基于确认切片输出修订正文、重点指南使用情况和修改说明。
6. 历史记录保存确认切片列表和本次回答引用状态。

收益：

- 用户可以在最终生成前把证据范围收紧。
- 减少模型把无关指南段落写进正文。
- 历史记录能解释“这次 AI 为什么这么改”。

## 前端交互设计

内容质量评审和 AI 整合共用一个切片确认组件，避免重复开发。用户需求分析暂不展示该组件，但组件设计应允许后续复用。

组件能力：

- 展示候选切片列表。
- 显示来源文件、参考数据源编号、标题路径、相关性理由。
- 支持展开查看前后文。
- 支持勾选、取消、全选高相关、清空。
- 支持按来源、标题路径、相关性排序。
- 支持用户手动搜索并补选切片。
- 显示已确认切片数量和总字数预估。

建议组件名称：

- 后端概念：`ReferenceChunkConfirmation`
- 前端组件：`ChunkConfirmationPanel`

不同环节传入不同的任务类型。第一版启用：

- `quality_review`
- `ai_integration`

后续预留：

- `user_needs`

## 后端接口设计

建议新增统一接口：

```http
POST /api/reference-chunks/search
```

请求：

```json
{
  "task_type": "quality_review",
  "disease": "糖尿病",
  "query": "诊断标准 空腹血糖 HbA1c",
  "reference_inputs": [
    {
      "id": 1,
      "filename": "2024 ADA 糖尿病指南.html",
      "text": "..."
    }
  ],
  "priority_reference_ids": [1],
  "limit": 30
}
```

响应：

```json
{
  "chunks": [
    {
      "chunk_id": "R1-C003",
      "source_id": 1,
      "source_filename": "2024 ADA 糖尿病指南.html",
      "title_path": "Diagnosis / Diagnostic Criteria",
      "text": "...",
      "context_before": "...",
      "context_after": "...",
      "source_ref_ids": ["15"],
      "score": 12.4,
      "reason": "命中章节标题、诊断标准、HbA1c"
    }
  ]
}
```

下游接口扩展：

- 内容质量评审接口增加 `confirmed_reference_chunks`。
- AI 整合接口增加 `confirmed_reference_chunks`。
- 用户需求分析接口第一版不扩展；后续如接入，再增加 `confirmed_reference_chunks` 或按需求主题保存的 `confirmed_chunks_by_topic`。

兼容规则：

- 如果前端没有传 `confirmed_reference_chunks`，后端沿用现有自动筛选逻辑。
- 如果传入确认切片，后端优先使用确认切片，不再从全文中额外选片。
- 保留 `reference_inputs`，用于构建引用锚点和兜底兼容旧流程。

## 检索策略

第一版建议采用轻量检索：

- 优先按 Markdown 目录标题切片，识别 `#`/`##`/`###` 以及解析后的 `[H1]`/`[H2]`/`[H3]` 等结构化标题。
- 标题切片保留完整 `title_path`，同级标题不能互相串联；例如 `## 诊断` 和 `## 治疗` 应是两个同级切片。
- 如果全文没有可识别标题，则按自然段切片；特别长的段落再按长度兜底拆分。
- 结合章节标题、任务 query、正文关键词做打分。
- 对指南、共识、年份较新的资料加轻微权重。
- 重点指南的切片优先展示。
- 没有命中时，返回每个重点指南的开头或核心片段作为兜底。

可选增强：

- 增加 BM25 或类似词频检索。
- 增加 LLM 重排：先召回 50 个候选，再让小模型选出最相关的 10-20 个。
- 增加人工反馈权重：用户多次确认的标题路径可以在同类任务中优先出现。

## 复杂向量库方案对比

复杂向量库是更完整的 RAG 基础设施。它会为每个切片生成 embedding 向量，把用户查询也转成向量，再用向量相似度召回语义接近的切片。

常见实现包括：

- PostgreSQL + pgvector。
- Qdrant。
- Milvus。
- Weaviate。
- FAISS。

向量库优势：

- 更擅长语义召回。比如用户 query 是“术后长期管理”，指南写的是“随访与监测”，即使关键词不同，也可能召回。
- 适合长期沉淀的大规模指南库。
- 可以跨疾病、跨主题查找语义相近证据。
- 可以和元数据过滤结合，例如只查某疾病、某年份、某机构、某证据等级。

向量库代价：

- 需要 embedding 模型，带来成本、延迟和模型选择问题。
- 需要维护索引生命周期，包括创建、更新、删除、重建。
- 需要处理权限隔离，避免用户看到不该看到的资料切片。
- 需要监控召回质量，否则错误召回更隐蔽。
- 需要处理切片版本一致性：指南更新后，历史任务引用的旧切片是否保留。
- 需要额外部署和运维向量数据库，或者依赖数据库扩展。

对当前项目的判断：

- 技术上完全可行。
- 但第一版不建议直接上复杂向量库。
- 当前场景中，用户上传的指南数量通常有限，且内容质量评审、AI 整合都有人审确认切片，轻量检索已经能解决主要问题。
- 如果未来指南库变成长期沉淀的大规模资料库，再升级向量库更合适。

## 演进路线

### 第一期：统一切片确认层

目标是让内容质量评审和 AI 整合能看到候选切片并确认。

实现内容：

- 抽取统一切片服务。
- 新增候选切片搜索接口。
- 新增前端切片确认组件。
- 内容质量评审和 AI 整合接入确认切片。
- 用户需求分析暂不接入，只保留任务类型和数据模型扩展空间。
- 下游 prompt 优先使用确认切片。

### 第二期：检索质量增强

目标是让候选切片更准，减少用户筛选负担。

实现内容：

- 引入 BM25 或更稳定的文本相关性算法。
- 增加 LLM 重排。
- 增加召回理由。
- 增加“用户补选切片”能力。
- 保存用户确认反馈，用于后续排序优化。

### 第三期：向量库或混合检索

目标是支持大规模指南库和更强语义召回。

实现内容：

- 为切片生成 embedding。
- 引入 pgvector、Qdrant 或其他向量索引。
- 建立关键词 + 向量 + 元数据过滤 + LLM 重排的混合检索。
- 增加索引版本和权限隔离。
- 建立召回质量评估集。

## 风险和处理

### 召回遗漏

风险：第一版轻量检索可能漏掉语义相关但关键词不同的片段。

处理：

- 支持用户手动搜索补选。
- 对重点指南保留兜底片段。
- 第二期加入 LLM 重排或向量召回。

### 用户确认负担增加

风险：多个环节都让用户确认切片，可能增加操作成本。

处理：

- 默认预选高相关切片。
- 提供“一键确认推荐切片”。
- 显示召回理由和标题路径，减少阅读成本。
- 第一版暂不在用户需求分析中加入确认步骤，避免把轻量分析流程变重。
- 对低风险任务允许沿用系统推荐。

### 引用标记混乱

风险：切片后丢失源内参考文献号，导致最终引用不可追溯。

处理：

- 切片时保留 `source_ref_ids`。
- 不改变原有参考数据源编号。
- 最终 prompt 明确只能使用切片中列出的引用标记。
- 响应继续返回 `reference_anchors`。

### 历史记录复现困难

风险：指南更新后，历史任务使用过的切片内容发生变化。

处理：

- 历史记录保存确认切片的文本快照。
- 切片增加来源文件名、source id、chunk id。
- 后续如建设长期库，再增加 source version 或 content hash。

## 测试

后端测试：

- 同一参考数据源能稳定切出带 `chunk_id` 的片段。
- 搜索接口能根据 query 返回相关切片。
- 重点指南切片优先展示。
- 无命中时返回兜底片段。
- 下游接口收到 `confirmed_reference_chunks` 时，只使用确认切片。
- 旧请求不传确认切片时，旧流程仍可运行。

前端测试：

- 切片确认组件能展示来源、标题路径、相关性理由。
- 用户可以勾选、取消、全选推荐、清空。
- 内容质量评审和 AI 整合能保存各自确认的切片。
- AI 整合历史能展示本次使用过的确认切片。
- 确认切片为空时，界面给出明确提示。

## 验收标准

- 内容质量评审前，用户能确认当前章节使用的指南切片。
- AI 整合前，用户能确认最终生成使用的指南切片。
- 用户需求分析第一版不要求确认指南切片，后续接入时可复用同一模型。
- 后端 prompt 中只包含用户确认过的切片，除非走旧流程兼容路径。
- 输出结果中的指南依据能追溯到确认切片。
- 没有确认切片时，系统不会悄悄把整篇指南全文发给最终 AI。
- 第一版不依赖向量库也能运行，但后续可以平滑升级到混合检索。

## 实施顺序

1. 抽取后端切片模型和切片构建函数，复用现有 AI 整合切片能力。
2. 新增统一候选切片搜索接口。
3. 新增前端 `ChunkConfirmationPanel` 组件。
4. 在 AI 整合接入确认切片，先打通最终生成链路。
5. 在内容质量评审接入确认切片。
6. 保存历史记录中的确认切片快照。
7. 补充后端和前端测试。
8. 根据真实使用反馈决定是否进入 BM25、LLM 重排或向量库阶段。
9. 后续如用户需求分析确实需要指南证据，再接入确认切片。
